# Задача: Визуальный редактор актов в Web UI (v2.0)

**Статус**: 📋 Запланировано  
**Приоритет**: 🟡 Средний  
**Категория**: Новые функции  
**Версия**: 2.0

## Описание

Сейчас для создания и редактирования актов пользователь должен вручную писать YAML-frontmatter в `.md` файлах. Это требует знания синтаксиса YAML, правил экранирования, структуры токенов `[[...]]` — порог входа высокий.

Цель — **визуальный редактор актов** прямо в Web UI, где пользователь работает с привычными формами: текстовые поля, календарь для дат, выпадающие списки. Приложение само формирует корректный YAML и сохраняет `.md` файл. Пользователю вообще не нужно открывать папку проекта или текстовый редактор.

## Текущий процесс (проблема)

1. Пользователь открывает папку workspace в Cursor/VSCode/блокноте
2. Создаёт новый `.md` файл
3. Пишет YAML-frontmatter вручную: `---`, `поля:`, `номер:`, отступы, кавычки
4. Экранирует спецсимволы (`"`, `\`, переносы строк)
5. Копирует структуру из другого акта, меняет значения
6. Если ошибка в YAML → непонятная ошибка валидации

## Целевой процесс (решение)

1. Пользователь открывает Web UI генератора
2. Нажимает «Редактор актов» → видит список актов проекта
3. Нажимает «Создать новый акт» или выбирает существующий
4. Открывается форма с полями:
   - Шаблон (dropdown)
   - Номер (prefix + value)
   - Имя файла (output_name)
   - Динамические поля: кнопка «Добавить поле» → название + значение
   - Для дат — календарь + выбор формата
   - Для длинных текстов — textarea
5. Нажимает «Сохранить» → `.md` файл создаётся/обновляется автоматически
6. Нажимает «Сгенерировать» → документ генерируется сразу из редактора

## Архитектура

### Навигация

Два «экрана» в SPA (single page application):

```
[Генератор]  ←→  [Редактор актов]
```

Переключение через кнопку/вкладку в шапке. URL-hash для навигации: `#generator`, `#editor`.

### Экран: Список актов

```
┌─────────────────────────────────────────────┐
│ Редактор актов              [+ Создать акт] │
├─────────────────────────────────────────────┤
│ 📄 акт-001-фундамент.md     [✏️ Ред.] [🗑] │
│ 📄 акт-002-утепление.md     [✏️ Ред.] [🗑] │
│ 📄 акт-003-кровля.md        [✏️ Ред.] [🗑] │
└─────────────────────────────────────────────┘
```

Каждая строка: имя файла, кнопки «Редактировать» и «Удалить».

### Экран: Редактор акта

```
┌─────────────────────────────────────────────┐
│ ← Назад к списку         акт-001.md        │
├─────────────────────────────────────────────┤
│ Шаблон:    [template.docx        ▼]        │
│ Имя файла: [3_АОСР_устр_дна_котлована   ]  │
│                                             │
│ ── Номер ──                                 │
│ Префикс:  [КЖ–Б-473–]                      │
│ Значение:  [3           ]                   │
│                                             │
│ ── Поля ──                          [+ Добавить поле]│
│                                             │
│ название_акта:  [Устройство дна котлована ]  │
│   [текст ▼]                                 │
│                                             │
│ объект:         [дом Б-473                ]  │
│   [текст ▼]                                 │
│                                             │
│ дата_начала:    [04.03.2026  📅]             │
│   [дата ▼]  формат: [d MMMMG yyyy г. ▼]    │
│                                             │
│ описание:                                   │
│   ┌──────────────────────────────────────┐  │
│   │ Многострочный текст...               │  │
│   │ Работы по [[объект]]                 │  │
│   └──────────────────────────────────────┘  │
│   [многострочный текст ▼]                   │
│                                             │
│ ── Недостающие поля шаблона ──              │
│ ⚠️ подрядчик, заказчик, инспектор           │
│                                             │
│ [💾 Сохранить]  [▶ Сгенерировать]           │
└─────────────────────────────────────────────┘
```

### Типы полей

| Тип | Виджет | Описание |
|---|---|---|
| **Текст** | `<input type="text">` | Однострочное текстовое поле |
| **Многострочный текст** | `<textarea>` | Для длинных описаний. Без YAML-экранирования — приложение само обернёт в `\|` или `>` |
| **Дата** | `<input type="date">` + dropdown формата | Календарь + выбор формата (dd.MM.yyyy, d MMMMG yyyy г., и т.д.) |
| **Ссылка на поле** | `<select>` из доступных полей | Удобный способ создать `[[токен]]` — выбрать поле из списка |
| **Число** | `<input type="number">` | Для числовых значений |

Тип поля выбирается из dropdown рядом с каждым полем. При смене типа виджет перерисовывается.

### Подсказка: недостающие поля шаблона

`docxtpl` умеет извлекать переменные из шаблона через `get_undeclared_template_variables()`. Редактор может показать:
- Какие переменные есть в шаблоне
- Какие из них уже заполнены
- Какие отсутствуют (⚠️ предупреждение)

Это избавит от ошибок «missing values for...» — пользователь видит что нужно заполнить прямо в редакторе.

### Наследование полей из `project.md`

Поля из `project.md` отображаются в редакторе акта как «наследуемые» (серый фон, подпись «из project.md»). Пользователь видит что уже заполнено на уровне проекта и может переопределить значение на уровне акта.

## API (бэкенд)

### Новые эндпоинты

```
GET  /api/act-data?workspace=...&act=...
POST /api/save-act
DELETE /api/delete-act
GET  /api/template-variables?workspace=...&template=...
GET  /api/project-fields?workspace=...
```

#### `GET /api/act-data`

Читает frontmatter акта и возвращает структурированный JSON:

```json
{
  "filename": "акт-001-фундамент.md",
  "template": "template.docx",
  "output_name": "3_АОСР_устр_дна_котлована",
  "number": {
    "prefix": "КЖ–Б-473–",
    "value": 3
  },
  "fields": {
    "название_акта": { "value": "...", "type": "text" },
    "описание": { "value": "Работы по [[объект]]", "type": "multiline" },
    "дата_начала": { "value": "04.03.2026", "type": "date", "format": "d MMMMG yyyy г." }
  }
}
```

#### `POST /api/save-act`

Принимает JSON из редактора, сериализует в YAML-frontmatter и записывает `.md` файл:

```json
{
  "workspace": "/path/to/workspace",
  "filename": "акт-001-фундамент.md",
  "is_new": false,
  "data": {
    "template": "template.docx",
    "output_name": "...",
    "number": { "prefix": "КЖ–Б-473–", "value": 3 },
    "fields": { ... }
  }
}
```

Ответственность бэкенда:
- Корректная YAML-сериализация (кавычки, переносы, спецсимволы)
- Валидация: имя файла, дубликаты, обязательные поля
- Атомарная запись (write + rename, чтобы не потерять файл при сбое)

#### `GET /api/template-variables`

Загружает `.docx` шаблон, извлекает все Jinja2-переменные:

```json
{
  "variables": ["название_акта", "объект", "дата_начала", "подрядчик", "заказчик"]
}
```

Используется для подсказок и автоматического создания недостающих полей.

#### `GET /api/project-fields`

Возвращает поля из `project.md` (наследуемые):

```json
{
  "fields": {
    "объект": "дом Б-473",
    "заказчик": "ООО Стройка"
  },
  "number": { "prefix": "КЖ–Б-473–" }
}
```

## Технические решения

### Фронтенд: фреймворк или vanilla JS?

Текущий проект — чистый vanilla JS без сборки. Для динамических форм с реактивным состоянием есть три варианта:

| Вариант | Плюсы | Минусы |
|---|---|---|
| **Vanilla JS** | Нет зависимостей, нет сборки, консистентность | Много boilerplate для реактивных форм, сложно поддерживать |
| **Alpine.js** (~15 KB) | Декларативный, без сборки, `x-for`, `x-model` из коробки | Новая зависимость (CDN), свой синтаксис |
| **Petite-Vue** (~6 KB) | Vue-like синтаксис, без сборки, реактивность | Ограниченная экосистема |
| **Preact + HTM** (~4 KB) | Компонентный подход, JSX-like без сборки | Непривычный для vanilla-проекта |

**Рекомендация**: Alpine.js — оптимальный баланс. Подключается через `<script>` (без npm/build), решает проблему реактивных форм (`x-model`, `x-for`, `x-show`), легко изучить. При этом текущий `app.js` для генератора может остаться на vanilla JS, а новый экран редактора — на Alpine.

### YAML-сериализация на бэкенде

Ключевая задача — генерировать чистый YAML из JSON, который приходит из формы:

```python
import yaml

def _yaml_representer_str(dumper, data):
    if "\n" in data:
        return dumper.represent_scalar("tag:yaml.org,2002:str", data, style="|")
    if any(c in data for c in ":#{}[]&*?|-><!%@`\"'"):
        return dumper.represent_scalar("tag:yaml.org,2002:str", data, style='"')
    return dumper.represent_scalar("tag:yaml.org,2002:str", data)

def serialize_act_to_yaml(data: dict) -> str:
    dumper = yaml.SafeDumper
    dumper.add_representer(str, _yaml_representer_str)
    return yaml.dump(data, Dumper=dumper, allow_unicode=True, default_flow_style=False, sort_keys=False)
```

Многострочные значения → YAML `|` блок. Спецсимволы → кавычки. Обычные строки → без кавычек.

### Атомарная запись файлов

Для предотвращения потери данных при сбое:

```python
import tempfile

def safe_write(path: Path, content: str):
    tmp = tempfile.NamedTemporaryFile(
        mode="w", encoding="utf-8",
        dir=path.parent, suffix=".tmp", delete=False,
    )
    tmp.write(content)
    tmp.close()
    Path(tmp.name).replace(path)
```

## Вопросы для проработки

### UX
- Автосохранение (debounce) или только по кнопке? Автосохранение удобнее, но может быть неожиданным.
- Показывать ли «diff» перед сохранением (что изменилось)?
- Нужен ли «предпросмотр YAML» для продвинутых пользователей (toggle «Показать исходник»)?

### Данные
- Как определять тип поля при загрузке существующего акта? Эвристики:
  - Если значение парсится как дата → тип «дата»
  - Если содержит `\n` → тип «многострочный»
  - Если содержит `[[...]]` → тип «ссылка»
  - Иначе → тип «текст»
- Нужно ли хранить метаданные типов полей (чтобы не терять при повторном открытии)?

### Шаблон
- `get_undeclared_template_variables()` может быть медленной для больших шаблонов — кэшировать?
- Как обрабатывать переменные с Jinja2-фильтрами в шаблоне (например, `{{ поле | upper }}`)?
